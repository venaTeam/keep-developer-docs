The **Workflows** service (`keep-workflows`) parses YAML workflow definitions, schedules their execution, and runs steps and actions through pluggable providers. It is a long-running FastAPI app with a singleton scheduler running in-process.

## Responsibilities

- Parse YAML workflow files into `Workflow` / `Step` objects.
- Match incoming events (alerts, incidents, manual triggers) against workflow trigger conditions (CEL).
- Schedule workflow runs and dispatch them to a `ThreadPoolExecutor`.
- Execute steps that **read** data (queries) and **act** on systems (notifications, tickets).
- Persist execution logs and surface them via the HTTP API for the UI.

## Entry points

The Workflows repository ships two entry points — pick deliberately.

| Entry point | Use it for | Notes |
| --- | --- | --- |
| `src/main.py` (`app` / `get_app()`) | Standalone Workflows service | Thin FastAPI shell. Mounts the `/workflows` router and starts the manager in `lifespan`. This is what production runs. |
| `src/api/api.py` | Embedded mode (full Keep backend) | Richer app with Arq worker integration, additional middlewares, and event-processing routes. Used when running Workflows inside the legacy monolith. |

`src/main.py` is the **canonical** standalone entry point.

## The `WorkflowManager` singleton

`src/workflowmanager/workflowmanager.py` is a `threading.Lock`-guarded singleton. On FastAPI startup the lifespan calls:

```python
workflow_manager = WorkflowManager.get_instance()
await workflow_manager.start()
```

`start()` boots `WorkflowScheduler`, which:

1. Polls the DB via `WorkflowStore` for workflows whose schedule is due.
2. Constructs `Workflow` objects via `Parser`.
3. Dispatches them to a `ThreadPoolExecutor` whose size is controlled by `WORKFLOWS_MAX_WORKERS` (default `20`).
4. Honors per-fingerprint execution strategy: `NONPARALLEL`, `NONPARALLEL_WITH_RETRY`, or `PARALLEL`.

Because the manager is a singleton, **tests that boot it directly must reset `WorkflowManager._instance = None` before and after**. The provided fixtures (`tests/fixtures/workflow_manager.py`) do this automatically.

## Scheduler internals

`WorkflowScheduler` (`src/workflowmanager/workflowscheduler.py`) is what actually runs your workflows. The mechanism:

- **Polling loop**. A background loop sleeps 1 second between iterations and calls `_handle_interval_workflows()` + `_handle_event_workflows()` every cycle. Every 100 iterations (~100s) it also runs `_timeout_workflows()` to mark stuck executions as ERROR.
- **Pool**. `ThreadPoolExecutor(max_workers=KEEP_MAX_WORKFLOW_WORKERS, thread_name_prefix="WorkflowScheduler")`. Default 20. `executor.submit()` returns a `Future`; the scheduler tracks them in `self.futures` with a callback to remove on completion.
- **Queue**. `self.workflows_to_run` is a plain Python `list` protected by `threading.Lock`. The producer (`WorkflowManager.insert_events`) appends; the consumer (`_handle_event_workflows`) atomically swaps the list out:
  ```python
  with self.lock:
      workflows_to_run, self.workflows_to_run = self.workflows_to_run, []
  ```
  This is **in-memory only** — a pod restart loses queued events, and the queue does not distribute across pods. A Redis-backed queue is a TODO in the code.

### Execution strategies

A workflow declares `strategy:` at the top level. The scheduler enforces it via `_get_unique_execution_number()`, which builds an integer key and uses it as a unique constraint when inserting the `WorkflowExecution` row. Conflict on the constraint → strategy decides what happens.

| Strategy | Key | Behaviour on collision |
| --- | --- | --- |
| `NONPARALLEL` | `SHA256(fingerprint:workflow_id) % MAX_SIGNED_INT` | `IntegrityError` → run is dropped, error logged. |
| `NONPARALLEL_WITH_RETRY` (default) | same | Re-enqueue with refreshed enrichments; retry next iteration. |
| `PARALLEL` | random UUID hash | No collision possible — multiple runs proceed concurrently. |

Use `NONPARALLEL_WITH_RETRY` unless you have a specific reason. `NONPARALLEL` is for expensive runs you'd rather drop than retry; `PARALLEL` is for genuinely independent steps where ordering does not matter.

### Timeouts

`WORKFLOWS_TIMEOUT` (default 120 minutes) is the wall-clock cap. `_timeout_workflows()` finds executions older than that and writes their status to `ERROR`. The thread keeps running — the timeout flag is informational, not a kill signal. If you ship long-running workflows, raise this and consider backing them with `KEEP_MAX_WORKFLOW_WORKERS` headroom.

## Triggers

Workflows declare what they react to with an `on:` clause. The supported trigger types and how each is dispatched:

| Trigger | How it fires |
| --- | --- |
| `alert` | `WorkflowManager.insert_events(tenant_id, [alerts])` evaluates each workflow's CEL filter against each alert. Match → enqueue. |
| `incident` | `WorkflowManager.insert_incident(...)`. Same matching mechanism, against `IncidentDto`. |
| `schedule` | Cron expression on the `Workflow` row. The scheduler's interval handler picks up workflows whose `next_run <= now`. |
| `interval` | Fixed delay in minutes. Computed as `last_run + interval`. |
| `manual` | `POST /workflows/{id}/run` on the HTTP surface. |

### CEL evaluation, with backward compat

`insert_events` evaluates the workflow's `on:` clause through `celpy`. Two helpers smooth over older syntax:

- `_convert_filters_to_cel` translates the deprecated regex-filter form to CEL: `r"prometheus|grafana"` becomes `source.contains("prometheus") || source.contains("grafana")`. Anything you find still using the old form should be migrated, but it works.
- `preprocess_cel_expression` normalizes severity comparisons. CEL has no enum type, so `severity > "warning"` would compare strings lexicographically (and silently produce wrong answers). The helper rewrites it against the numeric ordinal of `AlertSeverity`.

Two extra modifiers commonly used on alert triggers:

- `only_on_change` — fetch the previous alert with the same fingerprint via `get_previous_alert_by_fingerprint`. If nothing changed, skip the workflow.
- `severity_changed` — same idea but specifically for severity transitions.

Both modifiers turn the workflow into a state-change reactor rather than a heartbeat reactor — useful for "page on transition to critical" kinds of automations.

## The execution pipeline

For one workflow run:

```mermaid
flowchart LR
    YAML[workflow.yaml] --> Parse[Parser]
    Parse --> Wf[Workflow object]
    Wf --> Ctx[ContextManager]
    Ctx --> IO[IOHandler<br/>{{ ... }}]
    IO --> Step[Step]
    Step --> Provider[ProvidersFactory]
    Provider -->|read| Result[Step result]
    Result --> Next[Next step]
    Step --> Save[save_workflow_results]
```

1. **`Parser`** (`src/parser/parser.py`) — YAML to `Workflow` with two ordered lists: data-fetching `Step`s and side-effecting action `Step`s.
2. **`ContextManager`** (`src/contextmanager/contextmanager.py`) — holds event/alert/incident/consts/secret context.
3. **`IOHandler`** (`src/iohandler/iohandler.py`) — renders `{{ ... }}` against the context, with built-in helper functions.
4. **`Step`** — resolves a provider through `ProvidersFactory` and executes. Conditions and throttles gate action steps.
5. Results and errors are persisted via `save_workflow_results`.

## Premium providers

`PREMIUM_PROVIDERS = ["bash", "python", "llamacpp", "ollama"]` are blocked in multi-tenant mode. When adding a new provider with arbitrary code execution potential, update this guard.

## HTTP surface

`src/routes/workflows.py` is the primary router (mounted at `/workflows`). Every endpoint takes an `AuthenticatedEntity` via `IdentityManagerFactory.get_auth_verifier([scopes])`. Common scopes:

- `read:workflows`, `write:workflows`
- `read:providers`, `write:providers`
- `read:secrets`, `write:secrets`

`/workflows/providers` and `/providers` (root) are convenience endpoints that return everything the frontend needs to render the providers picker.

## Tests — gotchas

- `tests/conftest.py` builds a fresh SQLite file per `db_session` fixture and patches `src.common.core.db.engine`.
- `mocked_context_manager` is autouse — opt out explicitly when you need the real one.
- Both `test_app` and the manager fixture reset `WorkflowManager._instance = None`. Any test that boots the manager directly must do the same.
- Canonical YAML examples for parser tests live in `examples/workflows/` (47 files, good reference for valid shapes).

## Conventions specific to this repo

- `src/main.py:get_app()` re-imports routers lazily to avoid circulars and to pick up monkeypatched modules during tests. **Preserve this pattern** when adding routers.
- `src/common/core/db.py` is the **single source of truth** for query helpers. Don't add ad-hoc SQL in routes or the manager.
- The version lives in `pyproject.toml` and is mirrored in `src/main.py` (`KEEP_VERSION` default) and the Docker image tag pipeline. Bump all three together.
